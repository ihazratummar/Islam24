CONCRETE BLUEPRINT

1. The Room Database Entities

To support instant loading and dynamic translations, you need a scalable relational database structure. Instead of 4 separate JSON files, we use a structured SQL approach.

The Core Ayah Entity

This table holds the permanent Arabic text. It is populated instantly when the app installs via .createFromAsset().

@Entity(tableName = "ayahs", indices = [Index(value = ["surahNumber"])]) data class AyahEntity( @PrimaryKey val globalAyahNumber: Int, // 1 to 6236 val surahNumber: Int, val ayahNumberInSurah: Int, val arabicText: String, val transliteration: String )

The Dynamic Translation Entity

This table stores translations. By default, it might only contain English. When a user requests Bengali, you insert Bengali rows here.

@Entity( tableName = "translations", foreignKeys = [ForeignKey( entity = AyahEntity::class, parentColumns = ["globalAyahNumber"], childColumns = ["ayahId"], onDelete = ForeignKey.CASCADE )], indices = [Index(value = ["ayahId"]), Index(value = ["languageCode"])] ) data class TranslationEntity( @PrimaryKey(autoGenerate = true) val id: Int = 0, val ayahId: Int, // Matches globalAyahNumber val languageCode: String, // "en", "bn", "ur" val translationText: String )

The Unified Data Class (Relation)

Room allows you to query the Ayah AND its requested translation in one smooth operation.

data class AyahWithTranslation( @Embedded val ayah: AyahEntity, @Relation( parentColumn = "globalAyahNumber", entityColumn = "ayahId" ) val translations: List<TranslationEntity> )

2. The Room DAO

The Data Access Object defines how you interact with the database. Because it returns a Flow, your UI will update instantly and automatically as soon as a new translation finishes downloading.

@Dao interface QuranDao { // Get all Ayahs for a Surah, along with translations @Transaction @Query("SELECT * FROM ayahs WHERE surahNumber = :surahNumber ORDER BY ayahNumberInSurah ASC") fun getAyahsBySurah(surahNumber: Int): Flow<List<AyahWithTranslation>> // Check if a specific language is already downloaded @Query("SELECT COUNT(*) FROM translations WHERE languageCode = :langCode LIMIT 1") suspend fun isTranslationDownloaded(langCode: String): Int // Save new translations from the API to the database @Insert(onConflict = OnConflictStrategy.REPLACE) suspend fun insertTranslations(translations: List<TranslationEntity>) }

3. The Retrofit API (Dynamic Downloads)

We use the Al Quran Cloud API to fetch the full translation (e.g., Bengali) on-demand.

interface QuranApi { // Edition example: "bn.bengali" or "ur.jalandhry" @GET("quran/{edition}") suspend fun downloadTranslation( @Path("edition") edition: String ): Response<QuranApiResponse> } // The API Response Models data class QuranApiResponse(val data: QuranApiData) data class QuranApiData(val surahs: List<ApiSurah>) data class ApiSurah(val ayahs: List<ApiAyah>) data class ApiAyah(val number: Int, val text: String) // number is globalAyahNumber

4. Where and When to Call the API (The Logic)

This is the orchestration layer in your QuranRepositoryImpl.kt. This is where the magic happens.

The Flow of Events:

  1. User goes to settings and selects "Language: Bengali".
  2. ViewModel calls quranRepository.ensureTranslationExists("bn", "bn.bengali").
  3. Repository checks the Room Database. If Bengali exists, it does nothing!
  4. If it does not exist, it hits the API, maps the data, and saves it to Room.
  5. Because your Compose UI observes the Flow from the DAO, the Bengali text will magically appear on the screen the exact millisecond the database save is complete!
class QuranRepositoryImpl( private val api: QuranApi, private val dao: QuranDao ) : QuranRepository { override suspend fun ensureTranslationExists(langCode: String, editionId: String) = withContext(Dispatchers.IO) { // 1. Check if we already have this translation cached in Room val isDownloaded = dao.isTranslationDownloaded(langCode) > 0 if (isDownloaded) return@withContext // We already have it, do nothing! // 2. If not, fetch it from the API val response = api.downloadTranslation(editionId) if (response.isSuccessful && response.body() != null) { val newTranslations = mutableListOf<TranslationEntity>() // 3. Map the API response to our Room Entities response.body()!!.data.surahs.forEach { surah -> surah.ayahs.forEach { apiAyah -> newTranslations.add( TranslationEntity( ayahId = apiAyah.number, languageCode = langCode, translationText = apiAyah.text ) ) } } // 4. Save to database. This triggers the Flow to update the UI! dao.insertTranslations(newTranslations) } } }

5. Dynamic Settings (Font Size & Type)

To let users change the font size and style on the fly (just like Muslim Pro), you need to inject settings state into your AyahScreen without tightly coupling it.

The Plan: DataStore + StateFlow

  1. DataStore: Save user preferences (e.g., fontSize, fontFamilyType) using Jetpack DataStore.
  2. ViewModel: In your QuranViewModel, expose these settings as a StateFlow.
  3. UI Injection: Pass the settings down to your AyahScreen so the Composable automatically recomposes when settings change.
  4. The Settings Menu: Add an icon in your TopAppBar actions. When clicked, open a ModalBottomSheet containing a Slider for size and Radio Buttons for the font family.
// 1. Pass the dynamic settings into the Screen @Composable fun AyahScreen( ayahState: AyahState, dynamicFontSize: Int, // From Settings StateFlow dynamicFontFamily: FontFamily // From Settings StateFlow ) { // 2. Inside your AyahRow Text Text( text = ayahText, style = MaterialTheme.typography.headlineMedium.copy( fontFamily = dynamicFontFamily, fontSize = dynamicFontSize.sp, // CRITICAL: Line height must mathematically scale with font size! lineHeight = (dynamicFontSize * 2).sp, fontFeatureSettings = "calt, kern, liga, clig, ss01, ss03" ) ) }
